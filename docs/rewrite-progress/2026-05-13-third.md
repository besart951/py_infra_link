## Key Architectural Decisions

### Separation of concerns — auth vs. user identity
`UserCredential` is stored in a separate table from `User` and managed by a
separate `AuthModule`.  The `User` aggregate never holds a password hash.
`AuthModule` delegates user creation to `UserModule.create_user()`, avoiding
duplication of email/display-name validation logic.

### PasswordHasher port
Two adapters:
- `BcryptPasswordHasher` — bcrypt with gensalt, production-safe, ~72-char hashes
- `PlainPasswordHasher` — test-only, stores `"plain:<password>"`, fast, deterministic.
  The prefix prevents test hashes from accidentally being accepted by real bcrypt logic.

### jwt_codec.py
PyJWT's module-level `encode` and `decode` are accessed via a `__getattr__` proxy
in `jwt/__init__.py`, which pyright strict mode flags as "partially unknown".
Rather than scattering `# type: ignore` comments everywhere, a single helper
module wraps both calls, each with one isolated `# type: ignore` comment and
explicit return-type annotations.  All call sites are fully typed.

### JWT payload
```json
{ "sub": "<user_id as str>", "exp": <unix timestamp> }
```
`get_current_user` extracts `sub`, wraps it in `UserId(UUID(...))`, and fetches
the user from the database, raising 401 if the user no longer exists.

### Ownership guard
Project and notification routes check `current_user.id == UserId(path_param_id)`.
If the authenticated user is not the resource owner, a 403 Forbidden is returned
before the domain module is called.
