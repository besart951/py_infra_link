## Why This Candidate

Deletion test: if the shared kernel were deleted, `DomainError`, `Result`, `Page`, `Clock`, and typed IDs would leak directly into every module as ad-hoc exceptions and raw dicts.  The complexity reappears across all callers.  The kernel earns its keep.

Deletion test on database layer: if `atomic`, `get_session`, and `get_engine` were deleted, transaction control and engine configuration would be scattered across every route handler.  Complexity reappears everywhere.

---

## Dependency Classification

- **Shared kernel** (`errors`, `result`, `ids`, `clock`, `pagination`) — **In-process**.  Pure computation and in-memory state.  Always deepenable.  No external dependencies.
- **Database layer** (`engine`, `session`, `transaction`) — **Local-substitutable**.  PostgreSQL can be substituted with a test container or test database.  The `FixedClock` in the shared kernel is itself the second adapter that validates the `Clock` seam (production `SystemClock` + `FixedClock` for tests).

---

## Seam Decision

The `Clock` protocol is the only seam introduced.  It is a real seam because two concrete adapters are implemented:

1. `SystemClock` — production, delegates to `datetime.now(UTC)`
2. `FixedClock` — test, returns a deterministic value

All other shared kernel modules are sealed implementations with no seams.  The database layer exposes a flat interface (`get_engine`, `get_session`, `atomic`) with no protocol — the seam exists at the PostgreSQL level (production DB vs. test DB), not at the Python level.

## Interface Chosen

### Shared kernel interfaces (test surface)

| Module | Public interface |
|---|---|
| `errors` | `DomainError` hierarchy — raise and catch |
| `result` | `Ok[T]`, `Err[E]`, `Result[T, E]` — construct, match, unwrap |
| `ids` | `new_id(IdType)` — factory; all `NewType` aliases |
| `clock` | `Clock.now() -> datetime`; `SystemClock`; `FixedClock` |
| `pagination` | `PageParams` — validated params; `Page[T]` — paginated response; `Page.of(...)` |

### Database layer interfaces

| Module | Public interface |
|---|---|
| `engine` | `get_engine() -> AsyncEngine`; `dispose_engine()` |
| `session` | `get_session_factory()`; `get_session()` (FastAPI dep) |
| `transaction` | `atomic(session)` — async context manager |

### Design It Twice — `Result` type

Three designs were evaluated:

1. **Minimal** (`Ok | Err` dataclasses, `unwrap`, `is_ok`, `is_err`):  3 entry points.  Pattern-match friendly.  No `map`/`flat_map` — added only when callers need it.  **Chosen** — highest depth-to-interface ratio for current usage.

2. **Functional** (full `map`, `flat_map`, `fold`, `recover`):  Deeper but premature.  No callers yet — add when needed.

3. **Exception-only** (no `Result`, just `DomainError`):  Simpler at call site but loses explicit error paths in type signatures.  Rejected — domain errors must be visible in signatures.

### Current Problem

No domain Module existed yet, so there was no first-class User identity model, no domain Interface, no persistence model, and no HTTP endpoints beyond health.

### Proposed Solution

Create one deep User Module with a small application Interface:

- `create_user`
- `get_user`
- `list_users`

Back it with:

- SQLAlchemy Adapter (production)
- in-memory Adapter (tests)

and map domain errors at presentation boundary.

### Expected Benefits

- A reusable template for all remaining domains.
- Thin route handlers with domain logic in the Module.
- Explicit transaction usage in persistence writes.
- Interface-level tests stable under internal refactors.

### Dependency category

- User Module business logic: **in-process**
- SQLAlchemy persistence: **local-substitutable**

### Seam placement

`UserRepository` Protocol in `domain/interface.py` with two real Adapters:

1. `SqlAlchemyUserAdapter` (production)
2. `InMemoryUserAdapter` (tests)

### Why this increases Depth

Most caller complexity collapses into three entry points while validation, normalization, conflict checks, persistence mapping, and pagination behavior are hidden behind the Module Interface.

### Why this improves Locality

User behavior now lives under one module path (`modules/user/*`) with domain/application/infrastructure/presentation separation and no cross-cutting route-level business logic.

### Interface test surface

`UserModule` in `application/use_cases.py` is the test surface; tests call only its public methods.
