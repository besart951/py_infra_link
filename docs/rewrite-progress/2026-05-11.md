## Why This Candidate

Deletion test: if the shared kernel were deleted, `DomainError`, `Result`, `Page`, `Clock`, and typed IDs would leak directly into every module as ad-hoc exceptions and raw dicts.  The complexity reappears across all callers.  The kernel earns its keep.

Deletion test on database layer: if `atomic`, `get_session`, and `get_engine` were deleted, transaction control and engine configuration would be scattered across every route handler.  Complexity reappears everywhere.

---

## Dependency Classification

- **Shared kernel** (`errors`, `result`, `ids`, `clock`, `pagination`) ‚Äî **In-process**.  Pure computation and in-memory state.  Always deepenable.  No external dependencies.
- **Database layer** (`engine`, `session`, `transaction`) ‚Äî **Local-substitutable**.  PostgreSQL can be substituted with a test container or test database.  The `FixedClock` in the shared kernel is itself the second adapter that validates the `Clock` seam (production `SystemClock` + `FixedClock` for tests).

---

## Seam Decision

The `Clock` protocol is the only seam introduced.  It is a real seam because two concrete adapters are implemented:

1. `SystemClock` ‚Äî production, delegates to `datetime.now(UTC)`
2. `FixedClock` ‚Äî test, returns a deterministic value

All other shared kernel modules are sealed implementations with no seams.  The database layer exposes a flat interface (`get_engine`, `get_session`, `atomic`) with no protocol ‚Äî the seam exists at the PostgreSQL level (production DB vs. test DB), not at the Python level.

## Interface Chosen

### Shared kernel interfaces (test surface)

| Module | Public interface |
|---|---|
| `errors` | `DomainError` hierarchy ‚Äî raise and catch |
| `result` | `Ok[T]`, `Err[E]`, `Result[T, E]` ‚Äî construct, match, unwrap |
| `ids` | `new_id(IdType)` ‚Äî factory; all `NewType` aliases |
| `clock` | `Clock.now() -> datetime`; `SystemClock`; `FixedClock` |
| `pagination` | `PageParams` ‚Äî validated params; `Page[T]` ‚Äî paginated response; `Page.of(...)` |

### Database layer interfaces

| Module | Public interface |
|---|---|
| `engine` | `get_engine() -> AsyncEngine`; `dispose_engine()` |
| `session` | `get_session_factory()`; `get_session()` (FastAPI dep) |
| `transaction` | `atomic(session)` ‚Äî async context manager |

### Design It Twice ‚Äî `Result` type

Three designs were evaluated:

1. **Minimal** (`Ok | Err` dataclasses, `unwrap`, `is_ok`, `is_err`):  3 entry points.  Pattern-match friendly.  No `map`/`flat_map` ‚Äî added only when callers need it.  **Chosen** ‚Äî highest depth-to-interface ratio for current usage.

2. **Functional** (full `map`, `flat_map`, `fold`, `recover`):  Deeper but premature.  No callers yet ‚Äî add when needed.

3. **Exception-only** (no `Result`, just `DomainError`):  Simpler at call site but loses explicit error paths in type signatures.  Rejected ‚Äî domain errors must be visible in signatures.
