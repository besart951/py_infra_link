## Root Cause

With `@dataclass(frozen=True, slots=True)`:

```python
_users: dict[UserId, User] = field(default_factory=dict)
```

Pyright infers the field type from the `default_factory` callable's return type.
`dict` without type parameters returns `dict[Unknown, Unknown]`.  Even though the
field annotation `dict[UserId, User]` is present, pyright with `slots=True`
resolves the inferred type from the factory, producing `dict[Unknown, Unknown]`
which then conflicts with the annotation and raises `reportUnknownVariableType`.

The same issue applies to `list` (→ `list[Unknown]`) and `set` (→ `set[Unknown]`).

---

## Fix Applied

Replace bare `default_factory=dict/list/set` with a module-level typed factory
function for each adapter.  Pyright infers the field type from the function's
return annotation, which is fully concrete:

```python
def _user_store() -> dict[UserId, User]:
    return {}

@dataclass(frozen=True, slots=True)
class InMemoryUserAdapter:
    _users: dict[UserId, User] = field(default_factory=_user_store)
```

This approach:
- Is fully explicit — the intent is stated in the function signature
- Requires zero imports (no `cast`, no `TYPE_CHECKING` guards)
- Preserves the `frozen=True, slots=True` dataclass pattern unchanged
- Requires no changes to any call sites or tests

For `InMemoryHierarchyReader` (four mutable fields of mixed types):
```python
def _uuid_set() -> set[uuid.UUID]:
    return set()

def _uuid_list_map() -> dict[uuid.UUID, list[uuid.UUID]]:
    return {}
```

The two shared helper functions are reused for the three `dict` fields that all
have the same `dict[uuid.UUID, list[uuid.UUID]]` type.

## SQLAlchemy `rowcount` Fix

`notification/infrastructure/sqlalchemy_adapter.py` had three errors on the
`mark_all_as_read` method:

```
Cannot access attribute "rowcount" for class "Result[Any]"
Type of "rowcount" is unknown (reportUnknownMemberType)
Return type is unknown (reportUnknownVariableType)
```

`AsyncSession.execute()` is typed to return `Result[Any]` in SQLAlchemy's stubs,
which does not expose `rowcount`.  At runtime, DML statements return a
`CursorResult` which does have `rowcount`, but the stubs do not reflect this
distinction.

**Fix**: add `.returning(NotificationOrm.id)` to the `UPDATE` statement and count
the returned rows:

```python
stmt = (
    update(NotificationOrm)
    .where(...)
    .values(is_read=True)
    .returning(NotificationOrm.id)
)
result = await self._session.execute(stmt)
await self._session.flush()
return len(result.all())
```

This is cleaner than `rowcount`:
- Fully typed — `result.all()` returns `Sequence[Row[Any]]`, `len()` is always `int`
- More portable — some database drivers report `rowcount` as `-1`
- Consistent with the PostgreSQL `RETURNING` pattern already used in `mark_as_read`
