### 3. Fixed: `SqlAlchemyUserAdapter.create` used incompatible `atomic()` call

**Bug:** `atomic(session)` calls `session.begin()`, which raises
`InvalidRequestError: A transaction is already begun on this Session` after any
prior SQL (e.g. `get_by_email`) has triggered autobegin.

**Fix:** Replaced `async with atomic(self._session)` with
`async with self._session.begin_nested()` (SAVEPOINT).  `begin_nested()` always
works — it creates a SAVEPOINT when a transaction is already in progress, and
starts one first if none exists.  The SAVEPOINT is released on success or rolled
back on `IntegrityError`, leaving the session in a usable state.

Removed the now-unused `from app.database.transaction import atomic` import.

### 4. Fixed: All non-User ORM models used `TIMESTAMP WITHOUT TIME ZONE`

**Bug:** Nine ORM models used `Mapped[datetime]` (→ `TIMESTAMP` without timezone).
asyncpg raises a `DataError` ("can't subtract offset-naive and offset-aware
datetimes") when inserting a timezone-aware `datetime` (produced by
`SystemClock.now()` and `FixedClock`) into such a column.

**Fix:** Added `DateTime(timezone=True)` explicitly to `created_at` (and
`linked_at`) in all affected models:

- `FacilityOrm`, `BuildingOrm`, `ControlCabinetOrm`,
  `SpsControllerSystemTypeOrm`, `SpsControllerOrm`, `FieldDeviceOrm`,
  `BacnetObjectOrm`, `ProjectOrm`, `ProjectResourceLinkOrm`,
  `NotificationOrm`

These columns now map to `TIMESTAMP WITH TIME ZONE` in PostgreSQL, consistent
with `UserRecord` (which was already correct).
